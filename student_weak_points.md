## Критические проблемы (требуют усиленной практики)

### 1. Забывает $ перед переменными
**Проблема:** Пишет `{print 5}` вместо `{print $5}`, `"CHECK_SERVICE"` вместо `"$CHECK_SERVICE"`

**Почему важно:** Без $ это не переменная, а литеральная строка

**Как исправить:** 
- В awk колонки ВСЕГДА с $: `$1`, `$2`, `$5`
- В условиях переменные ВСЕГДА с $: `"$VAR"`, `$STATUS`
- Исключение: объявление переменной `VAR=value` (без $)

**Why:** Это базовый синтаксис bash, без этого скрипты не работают

**How to apply:** В каждом задании проверять использование переменных, делать акцент на $ в awk и условиях

---

### 2. Неправильный синтаксис sed
**Проблема:** Пишет `sed s//` вместо `sed 's/%//'`

**Почему важно:** Без кавычек и правильного синтаксиса sed не работает

**Правильный синтаксис:**
```bash
sed 's/old/new/'      # Заменить old на new
sed 's/%//'           # Удалить %
sed 's/old/new/g'     # Заменить все вхождения (global)
```

**Why:** sed — стандартный инструмент для обработки текста в production

**How to apply:** Давать больше задач с обработкой текста через sed

---

### 3. Забывает минус в сортировке
**Проблема:** Пишет `--sort=%cpu` вместо `--sort=-%cpu`

**Почему важно:** Без минуса сортирует от меньшего к большему (ascending), а нужно наоборот

**Правило:**
- `--sort=%cpu` — ascending (1, 2, 3...)
- `--sort=-%cpu` — descending (3, 2, 1...)

**Why:** В production всегда нужны топ процессы (самые нагруженные), а не самые лёгкие

**How to apply:** В каждом задании на мониторинг проверять направление сортировки

---

### 4. Не понимает tee полностью
**Проблема:** Думает, что `tee -a file.log` только пишет в файл, забывает про вывод в консоль

**Почему важно:** `tee` — это "тройник" (T-образная труба), раздваивает поток

**Правило:**
```bash
echo "test" | tee file.log       # Выводит в консоль И перезаписывает файл
echo "test" | tee -a file.log    # Выводит в консоль И дописывает в файл
echo "test" >> file.log          # Только в файл, без вывода в консоль
```

**Why:** В production скрипты должны логировать И показывать прогресс пользователю

**How to apply:** Давать задачи с логированием, объяснять разницу между tee и >>

## Некритические проблемы (мелочи, но нужно знать)

### 5. Лишний echo в подстановке команд
**Проблема:** Пишет `STATUS=$(echo "systemctl is-active nginx")` вместо `STATUS=$(systemctl is-active nginx)`

**Почему неправильно:** `echo "systemctl..."` просто выводит текст, а не выполняет команду

**Правило:**
```bash
# НЕПРАВИЛЬНО
STATUS=$(echo "systemctl is-active nginx")  # Вернёт строку "systemctl is-active nginx"

# ПРАВИЛЬНО
STATUS=$(systemctl is-active nginx)         # Выполнит команду и вернёт "active"
```

**Why:** Частая ошибка новичков — путать вывод текста и выполнение команды

**How to apply:** Объяснять разницу между echo (вывод текста) и $() (выполнение команды)

### 6. Усложняет простые задачи
**Проблема:** В задаче "проверь systemd-journald" написал скрипт с `$1` и переменными, хотя нужна была простая последовательность команд

**Почему проблема:** Overengineering — делает код сложнее, чем нужно

**Правило:** Если задача простая — решение должно быть простым. Не нужно писать универсальный скрипт для одноразовой проверки.

**Why:** В production простой код = надёжный код. Чем проще, тем меньше багов.

**How to apply:** Учить различать "написать скрипт" vs "выполнить задачу". Не всё требует скрипта.

**17. Не знал про fetch-depth: 0 для rollback**

### Неделя 3: Systemd

**1. Путаница Type=oneshot с Restart=always**

**Проблема:** Может попытаться использовать `Type=oneshot` с `Restart=always` (несовместимо!)

**Почему важно:** 
- `Type=oneshot` — процесс выполняется один раз и завершается
- `Restart=always` — перезапускать постоянно
- Это логически несовместимо

**Правило:**
- `Type=simple` + `Restart=always` — для долгоживущих процессов (while true)
- `Type=oneshot` + `Restart=no` — для разовых задач (скрипты)

**Why:** Частая ошибка новичков при создании systemd сервисов

**How to apply:** При создании сервисов проверять совместимость Type и Restart

---

**2. Забывает sudo systemctl daemon-reload**

**Проблема:** После изменения .service файла забывает перечитать конфигурацию

**Почему важно:** Без `daemon-reload` изменения не применятся

**Правило:** После ЛЮБОГО изменения .service или .timer файла:
```bash
sudo systemctl daemon-reload
```

**Why:** systemd кеширует конфигурацию, нужно явно перечитать

**How to apply:** Всегда напоминать про daemon-reload после изменения файлов

---

**3. Несколько OnCalendar создают несколько расписаний**

**Проблема:** Может написать:
```ini
OnCalendar=daily         # 00:00
OnCalendar=03:00         # 03:00
```
Думая, что это "переопределение", но на самом деле timer запустится **дважды** (в 00:00 И в 03:00)

**Правило:** Каждый `OnCalendar` добавляет новое расписание, а не заменяет предыдущее

**Why:** Неочевидное поведение systemd timers

**How to apply:** Объяснять, что несколько OnCalendar = несколько запусков

---

**4. Путаница с зависимостями (запускает database вместо frontend)**

**Проблема:** Если есть цепочка `frontend → api → database`, может запустить `database` и удивиться, что `api` и `frontend` не запустились

**Правило:** Зависимости работают **в обратную сторону**! Запускай **верхний уровень**, зависимости запустятся автоматически.

```bash
# ❌ Неправильно
sudo systemctl start database    # Запустится только database

# ✅ Правильно
sudo systemctl start frontend    # Запустятся: frontend → api → database
```

**Why:** Неинтуитивное поведение systemd зависимостей

**How to apply:** Объяснять, что нужно запускать "верхний" сервис, а не "нижний"

**5. Забывает local для переменных в функциях**

**Проблема:** Объявляет переменные в функциях без `local`, что делает их глобальными

**Why:** Без `local` переменная становится глобальной и может перезаписать внешние переменные

**6. Путает return и echo для возврата данных**

**Проблема:** Может попытаться вернуть данные через `return` вместо `echo`

```bash
# ❌ Неправильно
get_cpu() {
    local cpu=50
    return $cpu    # return только для exit code (0-255)!
}

# ✅ Правильно
get_cpu() {
    local cpu=50
    echo "$cpu"    # echo для данных
}
result=$(get_cpu)
```

**Правило:**
- `return` — только exit code (0-255)
- `echo` — для возврата данных

**Why:** Частая ошибка новичков при работе с функциями

**How to apply:** Объяснять разницу между return (exit code) и echo (данные)

**7. Неправильный порядок перенаправления (2>&1 > /dev/null)**

**Проблема:** Пишет `2>&1 > /dev/null` вместо `> /dev/null 2>&1`

```bash
# ❌ Неправильно (stderr пойдёт на экран)
```

**How to apply:** Объяснять порядок перенаправления: сначала stdout, потом stderr

**8. Забывает двоеточие в getopts для опций со значениями**

**Проблема:** Пишет `getopts "f"` вместо `getopts "f:"` для опции, требующей значение

```bash
# ❌ Неправильно (опция без значения)
while getopts "f" opt; do
    case $opt in
        f) file=$OPTARG ;;    # $OPTARG будет пустым!
    esac
done

# ✅ Правильно (опция со значением)
while getopts "f:" opt; do
    case $opt in
        f) file=$OPTARG ;;    # $OPTARG содержит значение
    esac
done
```

**Правило:** Двоеточие после буквы = опция требует значение

**Why:** Без двоеточия getopts не будет читать значение опции

**How to apply:** Проверять синтаксис getopts при работе с опциями

**9. Забывает declare -A для ассоциативных массивов**

**How to apply:** Всегда проверять наличие declare -A для ассоциативных массивов

**How to apply:** Запоминать команды для каждого дистрибутива

### 6. Путаница с ports vs expose

**Проблема:** Неправильно объяснил работу `ports` и `expose`

**Неправильное объяснение:**
> "если порт определен через expose (5000), а запущен через ports на другом (5001), то контейнер работать не будет"

**Правильное объяснение:**

**`ports`:**
- Публикует порт наружу (доступен с хоста)
- Формат: `"HOST_PORT:CONTAINER_PORT"`
- Пример: `"8080:5000"` — хост 8080 → контейнер 5000

**`expose`:**
- Только документирует порт (не публикует наружу!)
- Порт доступен **только другим контейнерам** в той же сети
- Не доступен с хоста

**Когда использовать:**
- `ports` — когда нужен доступ с хоста (веб-приложения)
- `expose` — когда нужен доступ только между контейнерами (база данных за Nginx)

**Why:** Это базовая концепция Docker networking

**How to apply:** Повторить Docker networking и разницу между внутренними и внешними портами

### 7. Непоследовательность docker-compose vs docker compose

**Why:** Непоследовательность выглядит непрофессионально

**How to apply:** Выбрать один вариант и использовать везде
