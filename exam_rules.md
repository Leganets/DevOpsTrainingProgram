# Правила создания экзаменов

## КРИТИЧЕСКИ ВАЖНО

**Студент пришёл учиться, а не получать халяву!**

От качества экзаменов зависит его будущее как DevOps-инженера. На собеседованиях не будут давать тесты с вариантами ответа — нужно **реально понимать** материал.

---

## ❌ ЗАПРЕЩЕНО

1. ❌ **Тесты с вариантами ответа (A, B, C, D)** — это халява!
2. ❌ **Правильные ответы под одним вариантом** (например, все под B)
3. ❌ **Простые вопросы типа "Что делает git status?"**
4. ❌ **Теоретические вопросы без практики**
5. ❌ **Вопросы, на которые можно ответить угадыванием**

---

## ✅ ОБЯЗАТЕЛЬНО

### Формат экзамена

**Количество вопросов:** 20-30 (не 60!)
- 10 вопросов на неделю (4 урока)
- 2-3 вопроса на урок

**Проходной балл:** 80% (как и раньше)

**Формат ответов:** Только **свободный ответ** или **код**

---

### Типы заданий

**1. Практические команды (40%)**

Напиши последовательность команд для выполнения задачи.

**Примеры:**
- "Напиши команды для создания ветки feature, переключения на неё, создания коммита и слияния в main"
- "Напиши команды для создания Docker образа myapp:v1.0 и запуска контейнера на порту 8080"
- "Напиши команды для настройки SSH ключа и добавления его в GitHub Secrets"

**Формат ответа:**
```bash
git branch feature
git checkout feature
# ... и так далее
```

---

**2. Исправление ошибок в коде (30%)**

Даю код с ошибками, студент должен найти и исправить.

**Примеры:**
- "Исправь ошибки в этом Dockerfile" (даю Dockerfile с 3-5 ошибками)
- "Исправь ошибки в этом GitHub Actions workflow" (даю workflow с ошибками)
- "Исправь ошибки в этом bash скрипте" (даю скрипт с ошибками)

**Формат ответа:**
```dockerfile
# Исправленный код
FROM python:3.9-alpine
# ...
```

---

**3. Написание кода с нуля (20%)**

Студент пишет код полностью сам.

**Примеры:**
- "Напиши Dockerfile для Python приложения с Flask"
- "Напиши GitHub Actions workflow для CI pipeline (lint, test, build)"
- "Напиши .dockerignore для Python проекта"
- "Напиши bash скрипт для проверки статуса сервисов"

**Формат ответа:**
```yaml
name: CI Pipeline
on:
  push:
    branches:
      - main
# ...
```

---

**4. Объяснение команд/концепций (10%)**

Студент объясняет, что делает команда или концепция.

**Примеры:**
- "Объясни, что делает команда `git reset --soft HEAD~1`. Что происходит с коммитом, staging area и working directory?"
- "Объясни разницу между `git rebase` и `git merge`. Когда использовать каждый?"
- "Объясни, зачем нужен `fetch-depth: 0` в GitHub Actions для rollback workflow"

**Формат ответа:**
```
git reset --soft HEAD~1 откатывает последний коммит, но оставляет изменения в staging area.
Это позволяет сразу сделать новый коммит без повторного git add.
```

---

## Структура экзамена

### Неделя (4 урока) = 20-25 вопросов

**Распределение:**
- 8-10 вопросов: Практические команды
- 6-8 вопросов: Исправление ошибок
- 4-5 вопросов: Написание кода
- 2-3 вопроса: Объяснение

**Сложность:**
- 30% — простые (базовые команды)
- 50% — средние (комбинации команд, исправление ошибок)
- 20% — сложные (написание кода с нуля, объяснение концепций)

## Примеры качественных вопросов

### Git

**Вопрос 1 (команды):**
Ты работаешь над feature веткой. Нужно временно переключиться на main, исправить hotfix, вернуться на feature и продолжить работу. Напиши последовательность команд.

**Ожидаемый ответ:**
```bash
git stash
git checkout main
# исправить hotfix
git add .
git commit -m "hotfix"
git checkout feature
git stash pop
```

---

**Вопрос 2 (исправление ошибок):**
Исправь ошибки в этих командах:
```bash
git checkout -b feature
git add app.py
git push origin feature
git checkout main
git merge main
```

**Ожидаемый ответ:**
```bash
git checkout -b feature
git add app.py
git commit -m "message"  # Забыл commit!
git push -u origin feature  # Забыл -u для первого push!
git checkout main
git merge feature  # Было git merge main (неправильно!)
```

---

### Docker

**Вопрос 3 (написание кода):**
Напиши Dockerfile для Python приложения:
- Базовый образ: python:3.9-alpine
- Рабочая директория: /app
- Установить зависимости из requirements.txt
- Скопировать код из app/
- Создать пользователя appuser
- Запустить python app/main.py

**Ожидаемый ответ:**
```dockerfile
FROM python:3.9-alpine

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app/ ./app/

RUN adduser -D appuser
USER appuser

CMD ["python", "app/main.py"]
```

---

**Вопрос 4 (объяснение):**
Объясни, зачем копировать requirements.txt отдельно перед копированием всего кода в Dockerfile?

**Ожидаемый ответ:**
```
Docker кеширует слои. Если скопировать requirements.txt отдельно и установить зависимости,
то при изменении кода (но не зависимостей) Docker использует кеш и не переустанавливает пакеты.
Это ускоряет сборку образа.

Правильный порядок:
1. COPY requirements.txt
2. RUN pip install (кешируется)
3. COPY . /app (код меняется часто, но кеш зависимостей сохраняется)
```

---

### GitHub Actions

**Вопрос 5 (исправление ошибок):**
Исправь ошибки в этом workflow:
```yaml
name: CI

on: push

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Run tests
        run: pytest tests/
```

**Ожидаемый ответ:**
```yaml
name: CI

on:
  push:
    branches:
      - main  # Указать ветку!

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code  # Забыли checkout!
        uses: actions/checkout@v4
      
      - name: Setup Python  # Забыли установить Python!
        uses: actions/setup-python@v5
        with:
          python-version: '3.9'
      
      - name: Install dependencies  # Забыли установить зависимости!
        run: pip install -r requirements.txt
      
      - name: Run tests
        run: pytest tests/
```

## Проверка ответов

### Критерии оценки

**Практические команды:**
- ✅ Правильная последовательность команд
- ✅ Все необходимые команды присутствуют
- ✅ Нет лишних команд
- ⚠️ Мелкие опечатки (например, забыл кавычки) — снять 0.5 балла
- ❌ Неправильная логика — 0 баллов

**Исправление ошибок:**
- ✅ Найдены все ошибки
- ✅ Исправлены правильно
- ⚠️ Найдены не все ошибки — частичный балл
- ❌ Не найдены критические ошибки — 0 баллов

**Написание кода:**
- ✅ Код работает
- ✅ Следует best practices
- ⚠️ Код работает, но не оптимален — частичный балл
- ❌ Код не работает — 0 баллов

**Объяснение:**
- ✅ Правильное объяснение с деталями
- ⚠️ Правильное, но неполное объяснение — частичный балл
- ❌ Неправильное объяснение — 0 баллов

---

## Обратная связь после экзамена

**После каждого вопроса:**
- ✅ Правильно — объяснить почему
- ❌ Неправильно — показать правильный ответ и объяснить ошибку
- ⚠️ Частично правильно — указать, что упущено

**В конце экзамена:**
- Итоговый результат (X/Y баллов, Z%)
- Разбор по темам (какие темы слабые)
- Рекомендации для повторения

---

## Why

Студент пришёл учиться, чтобы стать высококвалифицированным DevOps-инженером. От качества экзаменов зависит его будущее. На собеседованиях не будут давать тесты — нужно реально понимать материал и уметь применять на практике.

**Халявные экзамены = потеря времени студента = неуважение к его целям.**

---

## How to apply

1. **Перед созданием экзамена:** Прочитать этот файл
2. **При создании вопросов:** Использовать только разрешённые форматы
3. **При проверке ответов:** Давать детальную обратную связь
4. **После экзамена:** Указать слабые места и рекомендации

**НИКОГДА больше не создавать тесты с вариантами ответа!**
